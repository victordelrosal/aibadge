The Master Prompt

One copy-paste. The model interviews you with five short questions and then produces one well-formed SKILL.md inside a fenced code block.

Tested on GPT-5, Claude 4 Opus, and Gemini 2.5 Pro

Works on any capable chat model. The interview is deliberately context-free, so it fits any persona and any task.

You are the Skill Maker. Your job is to help me write one SKILL.md file: a small reusable procedure that one of my AI personas (or any AI assistant) can run on demand. A skill is not an identity. A skill is a recipe. We will build exactly one skill in this session, and we will build it well.

GROUND RULES

1. Ask the fewest questions possible. Five at most. One at a time. Wait for my answer before the next.
2. Do not lecture. Do not preview. Just ask.
3. The single most important thing in a skill is its trigger description: the line that tells an AI when to use this skill. We will spend extra care on it.
4. Output is a single markdown file inside one fenced code block, ready for me to save as a .md file.
5. No em dashes. Use commas, colons, semicolons, full stops, or parentheses.

THE INTERVIEW

Q1. Which persona on your team will own this skill? Give me the name and the archetype, or paste the persona file. If you do not have personas yet, just describe the kind of assistant you want to run this skill.

Q2. In one sentence, what is the task this skill should perform? Start with a verb. For example: "Draft a weekly funder scan", "Turn lecture notes into exam flashcards", "Plan dinners for the week from what is in the fridge", "Prepare me for a 1-to-1 with my report".

Q3. When should this skill fire? Give me three to five trigger phrases or situations you would actually use. The closer to your real words, the better. For example: "scan funders", "any new grants?", "Monday research", or "I have a 1-to-1 in an hour".

Q4. What does a great output of this skill look like? One short paragraph. Length, format (table, bullets, paragraphs), what is in it, what is left out. If a previous output exists that you liked, paste a sample.

Q5. What inputs does the skill need from me to do its job, and which of those should it ask me for if I forget to provide them? Three to five items maximum.

Once you have those five answers, do not ask anything else. Proceed to draft.

THE SKILL FILE FORMAT

Produce a single fenced markdown block in exactly this shape, populated with my answers. Be concrete. Every line should reflect my real task, not a generic placeholder.

```markdown
---
name: {short-kebab-case-id, e.g. weekly-funder-scan}
description: Use when {situation in my own words}. Triggers on {three to five real trigger phrases the user would say}.
owner: {persona name and archetype, or "any assistant" if no persona}
created: {today's date as YYYY-MM-DD}
---

# {Human-readable title}

## When to use
- Three to five concrete situations where this skill should fire.
- One line each.
- Include a "do not use when" line if there is a clear false-trigger to avoid.

## Inputs needed
- List the inputs the skill needs to do its job.
- For each, state whether it is required or optional, and what to ask the user if it is missing.

## Steps
A numbered list of the actual procedure, written so an AI assistant could follow it without further clarification. Each step is one short instruction. No more than seven steps. If the procedure is longer than seven steps, split it into a setup phase and an execution phase, each capped at seven.

## Output contract
- Length, format, and shape of the result.
- What must be in the output.
- What must not be in the output.
- One example skeleton (table headers, bullet shape, etc.) so the model knows what "good" looks like.

## One worked example
A short, real example of the skill in action. Inputs at the top. Output at the bottom. Five to ten lines total. This is the most important section: the example teaches the skill more than the steps do.

## Test it
A copy-pasteable line the user can say to a chat assistant (with the matching persona loaded) to verify the skill fires and runs correctly the first time.
```

NAMING

Choose the `name:` field as a short, lowercase, kebab-case identifier (verbs first when possible: weekly-funder-scan, draft-1to1-prep, flashcards-from-notes, dinner-from-fridge). Keep it short enough to remember, descriptive enough to find later.

CLOSING

After the fenced block, add a closing paragraph (no more than four sentences) telling me how to use this skill in practice: save it as `{name}.md` next to my persona file, paste the persona file plus this skill file into a new chat, then say one of the trigger phrases to test it. End by suggesting one related skill I might want to build next, based on what I described.

START NOW WITH Q1. NOTHING ELSE.

Detailed Step-by-Step Guide

From persona on a desk to a skill that fires on demand

Part 1. Persona is identity, skill is procedure

A persona is who: voice, archetype, defaults, how they push back. Built once, mostly stable. A skill is how: when to fire, the steps in order, what good output looks like. They sit on the same desk: one persona file plus many skill files, all plain markdown, all in the same workspace.

Think of a skill as the protocol you would hand a new colleague or intern. You do not need to micromanage every step. Modern AI is smart enough to fill the gaps. What it needs from you is the spine: input, process, output. Inputs: what does the skill need from me. Process: the steps in the order I want them. Output: the shape of the result, with one example so the model knows what good looks like.

Part 2. The anatomy of a SKILL.md

Every skill the master prompt produces has the same five sections. Knowing the anatomy makes editing fast: when something drifts, you know exactly which section to sharpen.

  • Front matter (name, description, owner, created). The description field is the trigger surface. This is where your real words live.
  • When to use. Three to five concrete situations, plus a "do not use when" line if a false trigger is likely.
  • Inputs needed. List what the skill needs and which inputs to ask for if you forget.
  • Steps. A numbered procedure, no more than seven. If it runs longer, split into setup and execution.
  • Output contract + worked example + test line. Length, format, what must and must not be in it. The worked example teaches the skill more than the steps do. The test line is one phrase you can paste into a fresh chat to verify it fires.

Part 3. Triggers in your real words

Triggers are the single most important field. A skill that never fires is worthless. The most common mistake is writing the trigger like a description instead of like the words you would actually say.

Bad: "a skill for funders". The model has nothing concrete to match. Good: "scan funders" · "any new grants?" · "Monday research". Three real phrases. The model recognises one and runs the skill.

Trigger on the work, not the worker. "scientist things" is too abstract; "any new grants?" is the actual sentence you would say to Aria.

Part 4. Iterate to a skill worth keeping

The first skill the model produces is rarely the right one. That is fine. Push back. In the recorded walkthrough I asked the model three times before landing on a skill worth saving:

  1. First pass: the model picked Attica the manager and produced a generic management skill. Functional, forgettable.
  2. Second pass: I said "create something more unique and particular." It moved to Milo the designer and built a skill about turning big taster ideas into 90 minute learner experiences. Better.
  3. Third pass: I said "assume the five agents have been hard at work over the past six months. One keeps getting asked to do something repetitive. Guess what." The model treated it as six month operational archaeology and produced harden-public-claims for Nora: turn copy full of unsupported claims into legally safer, evidence backed copy. That one was the keeper.

The "operational archaeology" prompt

If the model gives you a bland first draft, ask it to imagine the team has been running for six months. What task keeps recurring and creating drag? Solve that with a skill. Adds context the model needs to write something specific.

Part 5. Save it next to the persona

Open the workspace you built in Lesson 7. Inside, you already have a skills/ folder. Save the SKILL.md there using the kebab case name the model gave it. Examples: skills/harden-public-claims.md, skills/weekly-funder-scan.md, skills/dinner-from-fridge.md.

Save as .md, not .txt

The structure (front matter, headings, fenced examples) is what makes any AI tool read the file correctly. Saving as .txt defeats the format.

Part 6. Run it the slow way (web chat) and the fast way (terminal)

Both work. The difference is friction.

Web chat (ChatGPT, Claude.ai, Gemini)

  1. Open a fresh chat.
  2. Drop in (or paste) the persona file plus the SKILL.md.
  3. Tell the model: "Read the attached SKILL.md. When I say one of its trigger phrases, run that skill."
  4. Say a trigger phrase verbatim: "Aria, any new grants?"

Works every time, but you repeat the upload step in every new session.

Terminal (Claude Code, Codex, Gemini CLI)

  1. Open a terminal at your workspace root (the folder named after your team or organisation).
  2. Start the CLI tool of your choice: claude, codex, or gemini. They all read the folder.
  3. Ask it to scan all the agents and available skills in this folder. It walks the tree and reports back.
  4. Ask for a hypothetical scenario where the skill would fire, then say apply the skill.

No copy-paste. The persona and skill are already on the same desk; the AI tool just reads them. This is why a tidy workspace from Lesson 7 pays off here.

Part 7. Build a small library, not one giant skill

The temptation is to write one huge skill that handles everything. Resist it. Skills should be small, single purpose, and specific. One repeatable task per file.

Good starting backlog: pick the three or four tasks you do most often each week. One SKILL.md each. After a month you have a small library that compounds: any persona on your team can borrow a skill another persona uses. Skills travel across Claude, ChatGPT, and Gemini, so the library is portable too.

The shape of a working skill

Five answers in. One fenced markdown block out, well formed. Save next to the persona. Trigger phrase fires it. Output matches the contract. If any of those four steps wobbles, sharpen the section that owns it (description for triggers, output contract for shape, worked example for tone).

Troubleshooting

Common snags and fixes

The shape of a good run

Five short answers in. One fenced markdown block out, well-formed, ready to save. If you are getting anything else, the prompt was probably truncated when pasted. Re-copy from the button at the top of the code block, paste, and try again.

The model lectured me before asking Q1

It ignored Ground Rule #1. Reply: "Skip the preview. Ask Q1 only."

It asked more than five questions

Reply: "Stop. Re-read your ground rules. Five answers, then draft. No more questions."

The skill never fires when I say my trigger phrase

The trigger description is too generic, or your phrase does not match. Edit the description: line so it includes the exact words you would say. Three to five real phrases beats one abstract one.

The skill fires when I do not want it to (false trigger)

Add a "do not use when" line under ## When to use. Be specific about the lookalike situation that should NOT fire it.

The output drifts from one run to the next

Sharpen the ## Output contract. Add length limits, a required structure, and one example skeleton (table headers, bullet shape, etc.).

I got prose, not a fenced markdown block

Reply: "Re-output as a single fenced markdown code block in the SKILL.md format you were given."

The persona ignores the skill file when I paste both

Tell the model directly: "Read the attached SKILL.md. From now on, when I say one of its trigger phrases, run that skill." Some chat tools need an explicit instruction.

Common Mistakes to Avoid

  • Writing one giant skill that tries to do everything. Skills should be small. One task per file.
  • Using the archetype name as the trigger ("scientist things"). Triggers are about the work, not the worker.
  • Skipping the worked example. The example teaches the skill more than the steps do.
  • Saving as .txt. Save as .md so any AI tool reads the structure correctly.
  • Building skills before personas. The owner: field is what binds skills to who runs them.

The full walkthrough in text form

Why Skills

Now that we have created AI personas, agents that can help you on a day to day basis, it is time to give them skills: small repeatable recipes. Personas have personalities, voice, and context. Skills are the markdown files the agent runs in the same repeatable manner every time.

For example, if you have a persona like Aria the scientist, you have already defined how she thinks and her voice. A skill could be a weekly funder scan: how to perform the scan on a weekly basis, which sources to access, how to pull and format the information. Is the output a CSV file? What are the steps? That is what a skill is. It tells the agent when to run, what inputs to ask for, and the steps in order.

Think of it the same way you would teach a colleague or an intern. Anyone in the team would have these repeatable protocols. The important thing is you do not have to micromanage every last step. Modern AI is smart enough to figure out a lot. You just give a general idea of what you want, the inputs, the outputs, and the process. Input, process, output is a good way to think about it.

The Skill Maker Prompt

The base prompt for this lesson is the Skill Maker. Its job is to help you write one SKILL.md file. A skill, like a persona, is a markdown file: a small reusable procedure that one or more of your AI personas can run on demand.

If you copy the prompt and run it in a fresh instance, the first thing to do is reference the personas you already have. In ChatGPT's web interface, drop in the persona files, then paste the prompt. In this walkthrough I tweaked it slightly: instead of the prompt interviewing me, I told it to apply the prompt to one of the five agents, let the AI decide which, and then assume it interviews that agent. The output is a SKILL.md.

Iterating on the Skill

ChatGPT picked Attica the manager and produced a first draft. After reading it I thought, maybe there is something more interesting here. So I asked: actually, create a more unique and particular skill. It moved to Milo the designer and produced a skill about turning big taster ideas into vivid 90 minute learner experiences. Interesting, that fit Milo's role.

Notice the beauty: all the context the model needed came from the persona descriptions. I did not even include the description of the organisation. There was enough context in the role descriptions that the model could figure out what the team does and where they work.

Then I pushed once more: assume the five agents have been hard at work over the past six months. One of them keeps getting asked to do something repetitive and time consuming. Guess what that might be. Explain the pain points, then explain how a skill could solve it, then create that skill as a markdown file. The model treated it as six month operational archaeology: Nora kept being asked to turn academy claims into evidence backed, legally safer, commercially credible claims. The skill it produced: harden-public-claims. That one was a keeper.

Saving to the Workspace

If you recall the workspace structure from the previous lesson, you already have a skills/ folder. Save the skill there using the kebab case name the model gave it: harden-public-claims.md. Now if you open the skills folder, there it is, the same file you were just looking at in the chat.

This is where it can get tricky if you are not using the terminal. In a web chat, you copy and paste files in and out by hand. In the terminal, all of this is automatic. There is no copy paste. That is why the terminal interface is worth installing.

Running the Skill (Terminal)

From here on, the workspace folder is the root: in a real organisation it would be renamed from my-workspace-generic to the name of the organisation, in this example traders-academy. Open a new terminal at that folder. This time we will use Claude Code, just to show that skills work the same across Claude, Codex (ChatGPT), and Gemini. They are interoperable.

First prompt: scan all the agents and available skills in this folder. The agent walks the tree, lists the five personas and the one new skill, and reports there are no products defined. Good.

Then: create a hypothetical scenario where the skill would be used. Define the specific problem being faced. The model returns a Tuesday morning landing page launch, with a hero section that needs hardening: three reviewers will read the same paragraph within 48 hours, and each can block the project for a different reason. Concrete enough to test on.

Final step: apply the skill. The model is smart enough that you do not have to spell out the procedure. It runs the skill, produces an assumed claims table with risks and safer rewrites, and you have a usable output.

Comparing Web Chat and Terminal

For comparison, I ran the same scenario in the ChatGPT web interface, dropping in the same files. It still works: it detects the skill, recognises the other agents, produces a similar result. The point is not that the terminal is the only way. The point is the terminal makes this fast, organised, and findable. In the web interface you repeat yourself uploading files for every fresh session.

Skills Are the Unlock

Notice the arc: from a generic organisation, to a named one (Traders Academy), to a set of personas with voices and roles, to skills the personas can replicate on an ongoing basis. Recipes you can run over and over. This is a far more powerful way to work than re prompting from scratch every time.

And there are not dozens, not hundreds, but probably thousands of skills already published for Claude that also work for ChatGPT and Gemini. Skills are interoperable across systems, and skills are where a lot of the focus is right now in making AI agents work better, faster, and more efficiently on repetitive processes.
